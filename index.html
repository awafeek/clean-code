Clean Code

1 Naming

This an example of the how we can populate section of this guide, regarding naming I think we will have to build our rules and standards and start adding to this section.

As a starter I will post here some of the agreed consensus from chapter 1

  • Names should reveal the intention for the class, function or variable
  • Don't use list if it's not a list, always bear in mind technical and business names and use it for your favor to add more clarity not redundancy
  • Use Pronounceable names
  • Use Searchable names
  • Pick one word for concept, if you are using Save don't use store, persist or add for the same concept

2 Functions Checklist

This is the checklist we agreed about for smells while reviewing or writing functions, these smells is not a hard rock but you always need a good justification to break on of the items in this checklist:

  • Function should not exceeds 20 lines
  • Function should have maximum 2 arguments, 3 can be used with a strong reasoning
  • Function should not have output arguments
  • Function arguments shouldn't contain booleans for choosing between two behaviours
  • Command functions should not be used for query
  • Always return exceptions not error codes
  • Funtions shouldn't have more than 3 indent levels
